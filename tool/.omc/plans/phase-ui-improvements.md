## Issue A: Replace PhaseGate Stats Sidebar with File Browser

**Current**: Right sidebar (`~40%`) renders flat key-value pairs from `metadata` JSON.
**Target**: Show list of files generated during that phase. Clicking a file updates the main pane with its content (like the Files tab).

### Steps

**A1. Create phase-specific file listing API**
**New file**: `tool/src/app/api/engagements/[id]/phases/[phase]/files/route.ts`

- Accept engagement ID and phase number
- Use `listObjects()` to list files under the directories associated with that phase:
  - Phase 0: `research/`, `research/csv/`
  - Phase 1: `initial_questions/`, `claude-artefacts/tor-assessment*`
  - Phase 1A: `estimates/`, `claude-artefacts/technical-proposal*`, `claude-artefacts/optimistic*`
  - Phase 2: `claude-artefacts/response-analysis*`
  - Phase 3: `estimates/`, `claude-artefacts/estimate-review*`
  - Phase 3R: `claude-artefacts/gap-analysis*`, `claude-artefacts/revised*`
- Return `{ files: { key, name, dir, ext }[] }`

**A2. Update PhaseGate component**
**File**: `tool/src/components/phase/PhaseGate.tsx`

- Replace `stats` prop with `engagementId: string` and `phaseNumber: string`
- Right sidebar: fetch phase files from new API
- Render file list grouped by directory (reuse `DIR_LABELS` pattern from files page)
- On file click: fetch content via `/api/engagements/{id}/files/{path}` and display in main pane
- Add "Back to artefact" button to return to the PhaseArtefact markdown view
- Keep the version selector in the header

**A3. Update PhaseDetailPage to pass new props**
**File**: `tool/src/app/engagements/[id]/phases/[phase]/page.tsx`

- Remove `stats` computation (lines 125-128)
- Pass `engagementId={id}` and `phaseNumber={phase}` to PhaseGate instead of `stats`

### Acceptance Criteria
- [ ] Phase detail page right sidebar shows list of files generated by that phase
- [ ] Clicking a file in sidebar shows its content (markdown rendered, CSV in table, etc.) in the main pane
- [ ] "Back to artefact" button returns to the phase markdown output
- [ ] Files sidebar groups files by directory with appropriate labels

## Issue B: Clarity Breakdown on Dashboard

**Current**: `EngagementStats` component receives `clarityBreakdown` and renders it correctly with stacked bars and legend. The data comes from `/api/engagements/{id}/stats` which reads from `PhaseArtefact.metadata`. But the metadata extractor we added (`extractAssessmentMetadata`) only runs on new phase executions — existing Phase 1 artefacts have no metadata.

**Target**: Dashboard Requirement Clarity card reflects real breakdown data.

### Steps

**B1. Backfill metadata for existing Phase 1 artefacts**
**New file**: `tool/src/scripts/backfill-phase-metadata.ts`

- Query all PhaseArtefacts where `artefactType = TOR_ASSESSMENT` and `metadata IS NULL` and `contentMd IS NOT NULL`
- For each: run `extractAssessmentMetadata(contentMd)` and update the record
- Also backfill ESTIMATE artefacts with `extractEstimateMetadata`
- Idempotent — skip artefacts that already have metadata

**B2. No code changes needed to the stats API or EngagementStats component** — the data pipeline already works:
- `extractAssessmentMetadata` → saved to `PhaseArtefact.metadata` by phase-runner
- `/api/engagements/{id}/stats` → reads metadata and returns `clarityBreakdown`
- `EngagementStats` → renders stacked bar chart

### Acceptance Criteria
- [ ] After running backfill script, the Requirement Clarity card shows real numbers
- [ ] After running Phase 1 on a new engagement, clarity breakdown appears automatically

## Issue D: Conditional Site Audit in Phase 0

**Current**: Phase 0 prompt unconditionally instructs site audit "if a site URL is available."
**Target**: Skip site audit when engagement type is NEW_BUILD and TOR doesn't mention a current live site. Only perform site audit for MIGRATION or REDESIGN engagements.

### Steps

**D1. Update Phase 0 prompt to be engagement-type-aware**
**File**: `tool/src/lib/ai/phases/phase0-research.ts` (or `phase-prompts.ts`)

- Pass `engagementType` into the prompt builder
- Add conditional section:
  - If `engagementType === MIGRATION || engagementType === REDESIGN`:
    - Include full site audit instructions (current behavior)
  - If `engagementType === NEW_BUILD || engagementType === ENHANCEMENT`:
    - Replace site audit with: "Skip site audit — this is a new build. Focus on customer research, industry analysis, and TOR requirement analysis. Only inspect a site if the TOR explicitly provides a URL for an existing site that must be referenced."

**D2. Pass engagement type to phase config**
**File**: `tool/src/lib/ai/phases/index.ts` and `tool/src/workers/phase-runner.ts`

- Include `engagementType` in the phase config data passed to `getPhaseConfig()`
- The worker already has access to engagement data — just needs to pass it through

### Acceptance Criteria
- [ ] Phase 0 on a NEW_BUILD engagement skips site audit (no WebFetch calls for site inspection)
- [ ] Phase 0 on a MIGRATION engagement performs full site audit
- [ ] Phase 0 on a REDESIGN engagement performs full site audit
- [ ] Agent still performs site audit on NEW_BUILD if TOR explicitly provides a live site URL
